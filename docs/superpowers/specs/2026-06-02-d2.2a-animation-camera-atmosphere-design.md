# D2.2a — Animation + Camera + Atmosphere ("look & feel" pass) Design Spec

- **Date:** 2026-06-02
- **Status:** Approved (design); pending user spec review → implementation plan
- **Project:** Quassimodo — Phase D2.2a
- **Builds on:** the merged D2.1 minimal graphical match (`aplicacion`, PR #8). **Defers to D2.2b:** menu/GUI, user input, and interactive camera controls.
- **Companion docs:** [`ROADMAP.md`](../ROADMAP.md), [`KNOWN-BUGS.md`](../../../KNOWN-BUGS.md) (KB-D2-001..006), the D2.1 PR #8 compromise ledger, [`.wolf/cerebrum.md`](../../../.wolf/cerebrum.md), the D2 design [`2026-05-29-d2-graphical-port-design.md`](2026-05-29-d2-graphical-port-design.md).

## 0. Where this sits

D2.1 shipped a *minimal playable* graphical match: real board/pieces render, but pieces **teleport** (no animation), the camera is a loose fixed overhead, and there's no atmosphere (torches are static props, no shadows, no sky, flat ambient). D2.2 was split into two passes:

- **D2.2a (this spec) — the "look & feel" pass:** make the match *move and look* like a game — animated moves, a good camera, and atmosphere — with **zero user-input coupling**.
- **D2.2b (later) — the "interactivity" pass:** rebuild the menu/GUI, re-include input handling, and add interactive camera controls.

D2.2a stays on the lean argv-driven driver (`aplicacion agentA.py agentB.py`); it does **not** un-exclude the menu/`Aplicacion`/`ManejadorJuego`/`EventReceiver` files. No game-logic changes — `Reglas`/`Scripting`/the Juez-driven turn loop are untouched.

## 1. Goal & Non-Goals

### Goal
Turn the D2.1 still-life into a living match on macOS arm64 under IrrlichtMt 1.9.0mt15:
1. **Animated moves** — pawns **jump** from cell to cell and barriers **slide** into place, paced over time (the match no longer resolves in a few frames).
2. **A general animation system** — a reusable `Animador` (tween/animation manager) built to support not just the move-jump but **jump-over-opponent**, a **win/success** flourish, and **ambient** animations (torch flame), so future animation work is additive.
3. **A sim/presentation split via a command/event queue** — the match emits move-events; a presentation layer animates them, decoupled from the logical model.
4. **A good non-interactive camera** — a static, well-framed 3/4 perspective, behind a controller seam so D2.2b can add industry-standard orbit/pan/zoom without rework.
5. **Atmosphere** — billboard torch flames (KB-D2-001), piece shadows (KB-D2-002), a ground + sky backdrop, and basic lighting/tint.
6. **Cosmetic cleanup** — center the board base under the grid, strip the legacy debug `std::cout` spam (KB-D2-005).

### Non-Goals (→ D2.2b or later)
- Menu/GUI, message boxes, agent-selector UI.
- User input (keyboard/mouse) and **interactive** camera control — D2.2a only leaves the *seam*.
- Re-including `Aplicacion`/`ManejadorJuego`/`EventReceiver`/`PartidaConsola`.
- Game-logic / rules / agent changes.
- Networked/replay UI (the event queue *enables* replay/fast-forward, but no UI for it here).

## 1.5 Milestones (two PRs)

This slice is large, so it ships as **two milestones, each its own plan + PR**, each independently buildable and verifiable:

- **M1 — Movement & framing** (foundation): the event queue (`EventoJugada` + eager `MatchStepper` + `AnimPlayer`), the `Animador` with `SaltoAnim` (pawn jump) + `DeslizarAnim` (barrier slide), the dt-driver loop, restored `animacionesEnd` pacing, and the static framed camera + `CamaraController` seam. **Gate:** the AI-vs-AI match plays with visibly animated, time-paced moves to "Hay un ganador!". (No atmosphere yet — torches stay static props from D2.1; background may stay flat.)
- **M2 — Atmosphere & polish:** billboard torch flames (`LlamaAnim`, ambient) (KB-D2-001), blob piece shadows (KB-D2-002), ground quad + sky backdrop, board-base centering, and the debug-`cout` strip (KB-D2-005). **Gate:** the animated match now also looks "dressed" — flames, shadows, ground/sky, no void, no stdout spam.

M1 lands the architecture + the "feels like a game" win; M2 dresses it. M2 depends on M1. The atmosphere visuals in M2 are an explicit **first pass — expect a vision-review-and-iterate loop** on look (flame size/flicker, shadow softness, sky framing).

## 2. Architecture

The core move is a **simulation ↔ presentation split**, joined by a **command/event queue**, with a **frame-rate-independent (dt-based)** animation layer. (Rationale recorded in cerebrum: a turn-based board game has no continuous sim to interpolate — the *animation* is the interpolation — so the standard fit is sim/presentation decoupling + dt animation, not a fixed-timestep accumulator.)

```
                 ┌─────────────────────────── aplicacion_main.cpp (driver loop) ──────────────────────────┐
                 │  each frame:                                                                            │
                 │   dt = timer delta                                                                      │
                 │   matchStepper.pump(queue, maxInFlight)   // produce move-events (calls siguienteJugada)│
                 │   animPlayer.drain(queue)                 // turn new events into Animador animations   │
                 │   animador.update(dt)                     // advance all active animations by real dt   │
                 │   camera.update(dt)                       // (static now; seam for D2.2b controls)      │
                 │   beginScene → smgr->drawAll → endScene                                                 │
                 └─────────────────────────────────────────────────────────────────────────────────────────┘
   SIMULATION (logical, instant)                 │ queue │              PRESENTATION (visual, over time)
   PartidaGrafica/Partida + Reglas Juez ──emits──▶│Evento │──consumed──▶ AnimPlayer → Animador → Grafico nodes
   (Tablero model updated immediately)            │Jugada │              (Jugador/Barrera scene nodes tween)
```

### 2.1 The command/event queue (`EventoJugada`)
A small value type capturing what happened **logically**, carrying its own coordinates so the presentation never depends on how far the model has advanced:
```cpp
struct EventoJugada {
    enum Tipo { MOVIMIENTO, BARRERA, INICIO, GANADOR } tipo;
    int   jugador;                  // who
    core::vector3df desde, hasta;   // world coords for a MOVIMIENTO (from→to cell centres)
    // BARRERA: hasta = final slot, plus orientation;  GANADOR: jugador = winner id
    std::vector<int> celdaDesde, celdaHasta;  // grid coords (for jump-over detection etc.)
    Reglas::Direccion dir;          // barrier orientation
};
```
- **Producer** (`MatchStepper`, thin wrapper over `PartidaGrafica`): runs the **whole match eagerly** — it repeatedly calls the logical step (`siguienteJugada` → Juez → agent → Reglas model updated **immediately**) until a winner, pushing one `EventoJugada` per move into the queue. The queue is effectively **unbounded** (`maxInFlight = ∞`, the user-chosen default): the entire move stream is produced up front (in practice on the first frame, since Quoridor agents are synchronous and a match is short), then replayed as animation. The Grafico scene nodes are **not** touched during this logical run — only events are emitted. (The depth knob still exists for a future gated/buffered mode, but eager is the default.)
- **Consumer** (`AnimPlayer`): drains the queue **one blocking animation at a time** at real-time pace, converting each event into Animador animation(s) on the right scene nodes using the event's own `desde/hasta`. Pawn-move events animate the existing pawn node from `desde`→`hasta`; **barrier events create the `Barrera` scene node at drain time** and slide it in (so eager production doesn't require all barrier nodes up front). Because every event carries its own coordinates, replay is correct even though the Reglas model is already at the final state.

The Reglas model races to the end (eager); the presentation replays the recorded event stream over time. This is exactly why each event self-describes its coords — the animation never reads current model/node state.

**Failure note:** an eager logical run reaches an agent crash (e.g. KB-001 Minimax SIGSEGV) *before* any animation plays. The gate uses the proven clean pair (Camina vs Camina); other agents are out of scope. Acceptable.

### 2.2 The Animador (general animation manager)
A reusable, dt-based manager owning a list of active **Animacion** objects. Designed general from day one (the reason we chose this over ad-hoc per-class tweens):
```cpp
class Animacion {                       // base
public:
    virtual ~Animacion();
    virtual void update(u32 dtMs) = 0;  // advance by real elapsed time
    virtual bool terminada() const = 0; // done?
    virtual bool bloqueante() const { return true; } // gates the next move? (ambient = false)
};

class Animador {
public:
    void agregar(Animacion* a);         // take ownership
    void update(u32 dtMs);              // tick all; reap finished
    bool hayBloqueante() const;         // any blocking animation active? (the move gate)
    void limpiar();
private:
    std::vector<std::unique_ptr<Animacion>> activas;
};
```
Concrete animations for D2.2a:
- **`SaltoAnim`** — pawn jump: horizontal position lerp + a parabolic height arc (self-authored; standard `h*4*t*(1-t)` parabola), with an ease in/out on the horizontal. Blocking.
- **`DeslizarAnim`** — barrier slide-in: linear (eased) position lerp to the final slot. Blocking.
- **`LlamaAnim`** — torch flame: cycles the billboard's frame/scale/UV over time, loops forever. **Ambient (non-blocking).**
- Built to extend: a **jump-over** is a `SaltoAnim` with a higher arc / different target (detected from `celdaDesde/celdaHasta` adjacency to the opponent); a **win flourish** is a future `Animacion` (e.g. spin/scale on the winning pawn) — no new framework needed.

`PartidaGrafica::animacionesEnd()` is re-implemented as `!animador.hayBloqueante()` — restoring the 2010 move-pacing through the new manager.

### 2.3 Driver loop (replaces D2.1's instant-step loop)
```
once (first frame, or until the match ends):
  stepper.runToCompletion(queue)       // eager: produce the whole EventoJugada stream

each frame:
  u32 dt = device timer delta (real elapsed ms)
  if animador has no blocking animation and queue not empty:
      player.startNext(queue)          // pop the next event, build its Animador animation(s)
  animador.update(dt)                  // advance all tweens by dt (frame-rate independent)
  camera.update(dt)                    // no-op motion now; the seam for D2.2b
  render
  when the GANADOR event is reached → print "Hay un ganador! Jugador N" (+ future flourish)
```
Eager production + one-blocking-animation-at-a-time replay = the visible match paces at animation speed while the model is already final. Ambient animations (flames) run every frame regardless. (A future gated/buffered mode would interleave `runToCompletion` into the per-frame `pump`; the interface allows it without rework.)

## 3. Camera

A **static auto-framed 3/4 perspective**, computed from the board bounding box (like the D2.0 spike's self-framing) at a tasteful elevation/azimuth so the whole board + upright pawns read clearly — replacing D2.1's loose top-down framing (KB-D2-005). Wrapped in a tiny **`CamaraController`** seam:
```cpp
class CamaraController {
public:
    CamaraController(scene::ICameraSceneNode* cam, core::vector3df centro, f32 radio);
    virtual void update(u32 dtMs);   // D2.2a: holds the framed pose. D2.2b: orbit/pan/zoom.
    // D2.2b will add: onMouseDrag/onScroll for Maya/Blender-style orbit+zoom.
};
```
D2.2a ships only the static pose (no motion, no input). The seam means D2.2b drops in orbit/pan/zoom without touching the driver. (Optional, deferred: a *gentle auto-orbit* could be a one-line `update` later.)

## 4. Atmosphere (Milestone 2)

> **First pass — flagged for re-iteration.** The choices below are a sane starting point; after M2 renders, expect a vision-review-and-tune loop on the look (flame size/flicker rate, shadow softness/size, sky/ground framing and colors). Stencil shadows / shader lighting / fancier sky remain possible later upgrades.

All three "stripped subsystem" gaps get real replacements using surviving fork APIs:

- **Torch flames (KB-D2-001):** an **animated-texture billboard** per torch — `smgr->addBillboardSceneNode` (present in the fork) parented to the torch mesh, textured with the existing `fire.bmp`, animated by a `LlamaAnim` (ambient) cycling UV/scale for a flicker. No particles needed.
- **Piece shadows (KB-D2-002):** a **blob shadow** — a small dark textured quad (or flattened disc) under each pawn, following the pawn's `(x,z)` each frame. Cheap, reads as a shadow, no shadow-volume API. (Projected/stencil shadows are a possible later upgrade.)
- **Environment (KB-D2-005 / the "void"):** a **textured ground quad** under the board (the existing `piso3_TX.jpg`) + a **sky backdrop** — a large inward-facing textured dome/sphere mesh or a sky-colored billboard using `3.tree.skydome.png` (the fork has no `addSkyDomeSceneNode`/`addTerrainSceneNode`, so we build a simple mesh/quad). Replaces the flat clear-color background.
- **Lighting:** keep the correct flat-ambient approach from D2.1 (no dynamic lights in the fork), optionally add a subtle per-vertex tint / darker ambient for depth. No shader work in D2.2a.

## 8. Risks
- **R1 — Event-queue pacing/desync.** If the model runs ahead (maxInFlight>1) and the AnimPlayer animates from stale node state instead of the event's `desde/hasta`, pawns could jump from the wrong place. *Mitigation:* animations are driven **only** by the event's carried coordinates, never by current model state; default `maxInFlight` small.
- **R2 — Jump arc / cell-coordinate mapping.** The jump needs correct world from/to per cell; D2.1's `PartidaGrafica::getPosicionCelda` mapping + the board-base centering must agree. *Mitigation:* derive both from the same mapping; vision-verify a pawn lands centered on its target cell.
- **R3 — Billboard/quad APIs.** `addBillboardSceneNode` and manual textured quads must behave under the fork's GL2 path. *Mitigation:* spike each in the first atmosphere task; vision-check. (Low — billboards are core Irrlicht.)
- **R4 — Scope/size.** This slice is large. *Mitigation:* split into **two milestones / two PRs** (§1.5): M1 = movement & framing, M2 = atmosphere & polish. Each gets its own implementation plan; M1 ships and is reviewed before M2 starts.
- **R5 — Regression.** The PartidaGrafica change from "move node" to "emit event + animate" must not change *logical* outcomes. *Mitigation:* the Reglas model update path is unchanged (still `t->moverJugador`/`setBarrera`); only the *visual* update is rerouted through the queue. `consola` (no graphics) is unaffected and re-checked.

## 9. Success criteria
1. Animated AI-vs-AI match plays to a winner with visible pawn jumps + barrier slides, paced over time — verified by reading mid-tween screenshots.
2. A clean 3/4 framed camera; `CamaraController` seam in place for D2.2b.
3. Atmosphere present: flames, shadows, ground+sky; no void; no debug spam.
4. `Animador` + event queue are general (documented extension points for jump-over / success / replay).
5. `consola` regression passes; `otool` shows no `boost_python`.
6. PR(s) against `main` (repo-local `fabzter`; switch gh account for push, back after).

## 10. Out of scope (D2.2b and beyond)
- Menu/GUI/message boxes; user input; interactive camera controls (the seam is here, the controls are not).
- Stencil/projected shadows, shader lighting, particle-based flames (billboards/blobs are the D2.2a level).
- Replay/fast-forward UI (the queue enables it; no UI here).
- Re-including the menu-centric app layer.
