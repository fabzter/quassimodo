- **CRITICAL — Python embedding / linkage topology (study before touching the build):**
  - The app **embeds** CPython (`Py_Initialize` in `Scripting/InterpretePython.cpp`), then `import Reglas`.
  - `Reglas.so` (built by `src/AgenteWrapper`, hand-written Makefile) is a **Boost.Python extension** wrapping C++ classes (Tablero, Celda, Agente, AyudanteDeAgente, container_conversions, …). Agents `import Reglas` and subclass `Reglas.Agente`.
  - **Type-domain C++ classes live in `libReglas.so`** (single shared lib). Both the app (`Aplicacion`, `Grafico` link `-lReglas`) AND `Reglas.so` (links `-lReglas`) use it → RTTI/`type_info` single-sourced (required for cross-boundary casts).
  - **Cross-boundary object passing:** `ModuloPython.cpp` (in `libScripting`) does `object(boost::python::ptr(&tablero))` and wraps a `new AyudanteDeAgente` to inject `tablero`/`ayudante` into the agent namespace. Bidirectional: C++→Py (tablero/ayudante) and Py→C++ (agent instance via `boost::python::wrapper`/`AgentePythonWrapper`).
  - **HAZARD 1 — dual Boost.Python registry:** `Reglas.so` links boost_python **statically** (`-Bstatic -lboost_python -Bdynamic`) while `libScripting` links it **dynamically** (`-lboost_python-mt-py26`). Two registries → converters registered in the extension may be invisible to the embedder. The original got away with it on Linux via global symbol visibility / luck. **Modern fix: link boost_python as ONE shared lib everywhere; never static into the extension.**
  - **HAZARD 2 — implicit runtime symbol resolution:** `libScripting` uses `Reglas::` symbols but does NOT link `-lReglas`; it relies on Linux resolving undefined symbols at load time against the app's `libReglas.so`. macOS two-level namespace forbids this → must link deps explicitly (CMake `target_link_libraries`) or use flat namespace.
  - **No `sys.setdlopenflags(RTLD_GLOBAL)` anywhere** — Linux defaulted to making it work. macOS: build the extension as a MODULE with `-undefined dynamic_lookup`; resolve libpython/boost_python from the host process; share a single boost_python dylib. Plan must handle registry/RTTI sharing per-platform.

## Key Learnings (continued)

- **pyembed spike topology (2026-05-26):** `Boost::python` (FindBoost) does NOT automatically pull Python include dirs into extension module targets. Must also link `Python3::Module` on the extension (`Spike`) and `Python3::Python` on the embedder to get includes + libs. The `Python3::Module` target avoids linking libpython into the extension (correct for dlopen'd modules).
- **pyembed spike — import order (2026-05-26):** When embedding CPython and using a SEPARATE .so extension: the extension's `BOOST_PYTHON_MODULE` initializer runs at `import` time, registering type converters. You MUST call `import("Spike")` (or equivalent) in C++ BEFORE pushing any C++ Widget objects into Python namespace — otherwise `object(ptr(&widget))` throws "No Python class registered for C++ class Widget". The fix is to explicitly import the module from C++ right after extending `sys.path`.
- **pyembed spike — macOS rpath (2026-05-26):** No explicit rpath fix was needed. CMake's default `BUILD_RPATH` handling on macOS arm64 (in-tree build) correctly placed both `libwidget.dylib` and `Spike.so` so that dyld found them at runtime.
- **Irrlicht fork derivation (Task 7 — 2026-05-27):** `nix/irrlicht-fork.nix` builds IrrlichtMt from `github:minetest/irrlicht` rev `1.9.0mt15` (sha256-2zqG2JWdq+1QtgbgH8W7q+3yvREgMvINYh0phQvTv0c=). Canonical repo is `minetest/irrlicht` (luanti-org/irrlicht has no tags — redirects). Tags are `1.9.0mt0`..`1.9.0mt15`. The fork has a proper standalone CMake (C++17) with `install(TARGETS)` + header install to `include/irrlichtmt` + exported CMake config — no patches needed. **Darwin deps:** frameworks Cocoa/OpenGL/IOKit (the fork's `source/Irrlicht/CMakeLists.txt` does `find_library(COCOA_LIB Cocoa)/find_library(IOKIT_LIB IOKit)/find_package(OpenGL)`) PLUS `find_package(ZLIB/JPEG/PNG REQUIRED)` → add `zlib libjpeg libpng` to buildInputs on ALL platforms. nixos-24.11 still exposes `darwin.apple_sdk.frameworks.{Cocoa,OpenGL,IOKit}`. `BUILD_SHARED_LIBS` defaults ON (kept ON — matches what Luanti links). Set `-DBUILD_EXAMPLES=OFF`. Result: `result/lib/libIrrlichtMt.1.9.0.15.dylib` (+ `.dylib` symlink + lib/cmake/IrrlichtMt config) and 122 headers in `result/include/irrlichtmt/` incl. `irrlicht.h`, `IMeshLoader.h`, `ISceneManager.h`, `IGUIEnvironment.h`. - **IrrlichtMt CMake integration (Task 8 — 2026-05-27):** `find_package(IrrlichtMt CONFIG REQUIRED)` works perfectly in nix devShell. Imported target: `IrrlichtMt::IrrlichtMt`. A 640×480 window opens and runs 120 frames without issue.
- **D1.5 pybind11 spike — sys.path needs the executable's dir, not just the source dir (Task 5 — 2026-05-27):** The original spec/plan hardcoded `spikes/pybind` on `sys.path`, assuming that path holds both `agent.py` AND `PybindSpike.so`. With out-of-tree CMake builds, the extension lives in `build/spikes/pybind/`, NOT in the source `spikes/pybind/` — so `import PybindSpike` raises `ModuleNotFoundError` regardless of which CWD you run from. Fix: prepend the directory containing `argv[0]` (via `std::filesystem::path(argv[0]).parent_path()`) to `sys.path` IN ADDITION to the source spike dir. This same pattern will be needed for **D1′** when the real `Reglas.so` extension is rebuilt under pybind11 — the embedder (`InterpretePython`/console launcher) must add the runtime directory containing `Reglas.so` to `sys.path`, not assume CWD or `lib/`. The same gotcha was hidden in `spikes/pyembed/main.cpp` (works only if you happen to pass `argv[1]=build/spikes`); pybind11's `py::cast<Player*>()` runtime dispatch and `bind_vector` opaque mutation themselves all worked out-of-the-box with no shim needed (no `__delitem__`/`append` def's required — `bind_vector` exposes them natively for `std::vector<Cell*>`).
- **D1.5 §7 linkage invariants confirmed (Task 5 — 2026-05-27):** `otool -L build/spikes/pybind/pybind_spike` lists `@rpath/libpybindspike_domain.dylib` + `libpython3.11.dylib` (from nix store) and does NOT list `PybindSpike.so`. `otool -L build/spikes/pybind/PybindSpike.so` lists `@rpath/libpybindspike_domain.dylib` and does NOT list `libpython` (Python C-API resolves at load time via `-undefined dynamic_lookup` from the host process). This is the topology D1′ must preserve: extension and embedder share the SAME domain dylib (single RTTI source), extension never links libpython, embedder never links the extension.
